# 论文降重工具 UI 设计与接口说明文档

- 文档版本：`v1.0`
- 编写日期：`2026-03-20`
- 适用对象：`UI 设计师 / 产品经理 / 前端 / 后端`
- 对应需求文档：[论文降重工具需求文档](D:\projects by ai_coding\论文降重工具（1）\docs\product\论文降重工具需求文档.md)

## 1. 文档目的

本文档用于指导 Figma 原型与后续前后端实现，重点解决以下问题：

- UI 整体应该做成什么样
- 页面该包含哪些核心模块
- 每个模块有哪些状态和交互
- 页面和接口之间如何对应
- 前后端在字段和流程上如何对齐

这份文档优先服务于一期产品设计，产品定位为 `个人论文助手`，`Web 端优先`，面向论文上传、检测、改写、对比、复检和导出全流程。

## 2. 设计总方向

### 2.1 设计关键词

- 学术感
- 专业感
- 清晰可控
- 风险可视化
- 不花哨但不死板

### 2.2 整体风格建议

UI 不建议做成通用聊天机器人风格，也不建议做成工具后台式的粗糙表格界面。建议采用“学术工作台”方向：

- 页面基底以浅色为主，突出信息密度和可读性
- 风格偏专业、理性、可信，不要娱乐化
- 用少量强调色区分风险等级和关键动作
- 核心价值不是“AI 很强”，而是“结果可解释、可对比、可回退”

### 2.3 推荐视觉方向

- 主色：深蓝或蓝绿色，体现专业和稳定
- 风险色：
  - 低风险：绿色
  - 中风险：橙色
  - 高风险：红色
  - AIGC 风险：可使用偏洋红或橙红系，但不要和重复率风险完全混淆
- 背景色：偏冷白或浅灰白
- 卡片：大圆角轻阴影，保持整洁
- 图表：环形图、条形图、热区标签图即可，不需要复杂 3D 图形

### 2.4 字体建议

Figma 出图时建议：

- 中文主字体：`思源黑体`、`HarmonyOS Sans SC`、`阿里巴巴普惠体`
- 英文数字：可搭配 `IBM Plex Sans` 或 `Inter`
- 标题字重偏高，正文保持中等字重，避免过多粗体造成压迫感

### 2.5 间距与圆角建议

- 主页面左右留白：`32px`
- 卡片内边距：`20px` 或 `24px`
- 模块间距：`16px` 或 `20px`
- 按钮高度：`40px` / `44px`
- 输入框高度：`40px`
- 卡片圆角：`16px`
- 弹窗圆角：`20px`

1. 首页 / 工作台

3. 检测报告页

顶部状态栏建议包含：

- 当前论文名称
- 当前版本
- 全局搜索入口
- 最近任务状态
- 用户菜单

## 5. 页面详细说明

## 5.1 首页 / 工作台

### 5.1.1 页面目标

让用户快速开始一次论文处理任务，看到最近的论文、最近的检测和最近的改写进度。

### 5.1.2 页面布局建议

推荐采用三段式布局：

- 顶部：欢迎区 + 主行动按钮
- 中部：最近论文 / 最近任务
- 底部：快捷功能入口

### 5.1.3 页面模块

#### 模块 A：欢迎区

包含：

- 页面标题：`论文工作台`
- 一句说明：`上传论文、检测风险、逐段改写并复检`
- 主按钮：`上传新论文`
- 次按钮：`进入模型配置`

#### 模块 B：统计概览

建议 4 张数据卡片：

- 我的论文数
- 待处理高风险段落数
- 今日检测任务数
- 今日改写采纳数

#### 模块 C：最近论文列表

每张论文卡片展示：

- 论文标题
- 最后更新时间
- 当前版本号
- 最近一次风险概览
- 快捷按钮：`查看详情`、`继续改写`

#### 模块 D：最近任务

分成两个 Tab：

- 检测任务
- 改写任务

展示字段：

- 任务名称
- 类型
- 状态
- 开始时间
- 结果摘要
- 操作入口

#### 模块 E：快捷入口

图标卡片式入口：

- 上传论文
- 开始检测
- 改写高风险段
- 复检
- 导出报告

### 5.1.4 页面状态

- 空状态：提示用户先上传第一篇论文
- 加载状态：卡片骨架屏
- 错误状态：顶部提示条 + 重试按钮

## 5.2 论文详情页

### 5.2.1 页面目标

让用户确认论文结构解析结果，决定哪些内容纳入后续检测和改写。

### 5.2.2 页面布局建议

建议采用 `左侧结构目录 + 中间正文预览 + 右侧配置栏` 三栏布局。

### 5.2.3 页面模块

#### 模块 A：文件信息区

展示：

- 论文标题
- 文件格式
- 文件大小
- 上传时间
- 当前版本

#### 模块 B：结构目录

展示：

- 一级标题
- 二级标题
- 特殊区域识别

支持点击目录定位到正文。

#### 模块 C：正文预览

展示解析后的正文段落，支持：

- 高亮当前选中段
- 显示段落编号
- 展示章节归属

#### 模块 D：排除范围设置

支持以下开关：

- 排除封面
- 排除目录
- 排除参考文献
- 排除附录

还要支持“手动框选排除段落”。

### 5.2.4 页面主按钮

- `保存解析结果`
- `开始检测`

### 5.2.5 页面状态

- 解析中
- 解析完成待确认
- 解析失败

## 5.3 检测报告页

### 5.3.1 页面目标

把“风险是什么、风险在哪、先处理什么”讲清楚。

### 5.3.2 页面布局建议

建议采用 `顶部总览 + 中部风险分布 + 下部段落列表` 布局。

### 5.3.3 页面模块

#### 模块 A：报告总览

展示 3 到 4 个核心指标卡：

- 全文重复风险分
- 全文 AIGC 风险分
- 高风险段落数量
- 已处理段落数量

#### 模块 B：风险分布图

建议使用：

- 环形图：重复风险占比
- 环形图：AIGC 风险占比
- 条形图：章节风险分布

#### 模块 C：风险筛选栏

支持筛选：

- 全部段落
- 仅高风险
- 仅重复风险
- 仅 AIGC 风险
- 仅未处理
- 按章节筛选

#### 模块 D：高风险段落列表

每个段落卡片包含：

- 段落编号
- 所属章节
- 原文摘要
- 风险分
- 风险类型
- 风险说明
- 建议动作
- 操作按钮：`加入改写`、`查看详情`

#### 模块 E：段落详情抽屉

点击段落后在右侧抽屉展示：

- 原文完整内容
- 风险说明
- 证据文本
- 历史处理记录
- 快速发起改写

### 5.3.4 页面重点

- 风险必须一眼可见
- 不要只给总分，要能定位到段
- 高风险段操作入口必须明显

## 5.4 改写对比台

### 5.4.1 页面目标

让用户精确比较原文和改写稿，并做出“采纳 / 回退 / 重写”的决策。

### 5.4.2 页面布局建议

建议采用 `顶部工具栏 + 左原文 + 右改写 + 底部批量操作区` 布局。

### 5.4.3 页面模块

#### 模块 A：顶部工具栏

展示：

- 当前论文名称
- 当前段落位置
- 使用的改写策略
- 使用的模型
- 重新改写按钮
- 批量改写按钮

#### 模块 B：左侧原文区

展示：

- 原文内容
- 改写前风险分
- 原文锁定提示

#### 模块 C：右侧改写区

展示：

- 改写内容
- 改写说明
- 改写后风险分
- 改写风格标签

#### 模块 D：差异高亮区

需要支持：

- 插入高亮
- 删除高亮
- 改写高亮
- 仅看差异

#### 模块 E：候选版本区

同一段可有多个候选版本，展示：

- 版本 1 / 版本 2 / 版本 3
- 不同版本的风险变化
- 当前是否采纳

#### 模块 F：底部操作区

按钮建议：

- `采纳当前改写`
- `回退到原文`
- `再次改写`
- `标记稍后处理`
- `复检当前段`

### 5.4.4 页面重点

- 原文和改写一定要并排对照
- 分数变化必须清晰
- 用户必须能安全回退

- 模型配置为空

## 7. 交互流程说明

## 7.1 核心流程

1. 用户从工作台点击 `上传新论文`
2. 上传后进入论文详情页
3. 确认解析结果并设置排除范围
4. 点击 `开始检测`
5. 进入检测报告页查看高风险段
6. 选择一个或多个段落加入改写
7. 进入改写对比台
8. 采纳部分改写结果
9. 点击 `复检`
10. 查看变化后导出结果

## 7.2 页面跳转关系

- 工作台 → 上传弹窗 → 论文详情
- 论文详情 → 检测报告
- 检测报告 → 改写对比台
- 改写对比台 → 复检结果
- 任意页 → 模型配置页

## 8. 接口设计原则

### 8.1 通用规则

- 接口风格建议使用 REST
- 返回统一响应结构
- 任务型接口采用异步任务模式
- 长耗时任务需要轮询或订阅状态

### 8.2 统一返回结构

```json
{
  "code": 0,
  "message": "ok",
  "data": {},
  "requestId": "req_xxx"
}
```

失败示例：

```json
{
  "code": 40001,
  "message": "模型连接失败",
  "data": null,
  "requestId": "req_xxx"
}
```

## 9. 主要接口清单

## 9.1 文档相关接口

### `POST /api/documents/upload`

用途：上传论文文件

请求：

- `file`
- `title`
- `language`

返回核心字段：

- `docId`
- `title`
- `sourceFile`
- `version`

### `GET /api/documents/{docId}`

用途：获取论文详情

返回核心字段：

- `docId`
- `title`
- `language`
- `sections`
- `paragraphs`
- `version`

### `POST /api/documents/{docId}/parse`

用途：重新解析论文结构

请求字段：

- `excludeCover`
- `excludeCatalog`
- `excludeReferences`
- `excludeAppendix`

### `PATCH /api/documents/{docId}/exclusions`

用途：保存排除范围设置

## 9.2 模型配置接口

### `GET /api/model-providers`

用途：获取模型列表

### `POST /api/model-providers`

用途：新增模型配置

请求示例：

```json
{
  "providerType": "local",
  "name": "本地 Ollama",
  "baseUrl": "http://127.0.0.1:11434",
  "apiKey": "",
  "model": "qwen2.5:7b",
  "capabilities": ["rewrite", "detect"],
  "timeoutMs": 60000,
  "concurrency": 2,
  "contextWindow": 8192
}
```

### `PUT /api/model-providers/{providerId}`

用途：更新模型配置

### `DELETE /api/model-providers/{providerId}`

用途：删除模型配置

### `POST /api/model-providers/{providerId}/test`

用途：测试连接

返回：

- `status`
- `latencyMs`
- `message`

### `POST /api/model-providers/{providerId}/set-default`

用途：设置默认模型

请求：

```json
{
  "scene": "rewrite"
}
```

`scene` 可选值：

- `rewrite`
- `detect`

## 9.3 检测接口

### `POST /api/detection-tasks`

用途：发起检测任务

请求示例：

```json
{
  "docId": "doc_001",
  "taskTypes": ["plagiarism", "aigc"],
  "providerId": "provider_001"
}
```

返回：

- `taskIds`
- `status`

### `GET /api/detection-tasks/{taskId}`

用途：获取检测任务状态和结果

返回核心字段：

- `taskId`
- `taskType`
- `status`
- `summaryScore`
- `segmentResults`

### `GET /api/documents/{docId}/detection-summary`

用途：获取某篇论文最新检测概览

返回核心字段：

- `plagiarismScore`
- `aigcScore`
- `highRiskCount`
- `processedCount`

## 9.4 改写接口

### `POST /api/rewrite-tasks`

用途：发起改写任务

请求示例：

```json
{
  "docId": "doc_001",
  "segmentIds": ["seg_1", "seg_2"],
  "strategy": "lower_aigc",
  "providerId": "provider_001",
  "model": "qwen2.5:7b",
  "options": {
    "tone": "academic",
    "preserveTerms": true,
    "preserveCitations": true,
    "preserveNumbers": true
  }
}
```

### `GET /api/rewrite-tasks/{taskId}`

用途：获取改写任务进度和结果

返回核心字段：

- `taskId`
- `status`
- `candidates`
- `progress`

### `POST /api/rewrite-candidates/{candidateId}/accept`

用途：采纳某个候选版本

### `POST /api/rewrite-candidates/{candidateId}/rollback`

用途：回退某个已采纳版本

### `POST /api/rewrite-tasks/{taskId}/retry`

用途：再次改写

## 9.5 复检接口

### `POST /api/recheck-tasks`

用途：对已采纳内容重新检测

请求：

```json
{
  "docId": "doc_001",
  "segmentIds": ["seg_1", "seg_2"]
}
```

### `GET /api/recheck-tasks/{taskId}`

用途：获取复检结果

返回核心字段：

- `beforeScores`
- `afterScores`
- `changedSegments`

## 9.6 导出接口

### `POST /api/exports`

用途：创建导出任务

请求示例：

```json
{
  "docId": "doc_001",
  "exportType": "diff_report"
}
```

`exportType` 可选值：

- `final_doc`
- `diff_report`
- `summary_report`

### `GET /api/exports/{exportId}`

用途：获取导出结果

## 13. 文案与交互语气建议

产品语气建议偏专业、清晰、支持型，不要像营销页，也不要像命令行报错。

示例：

- 好文案：`检测完成，发现 12 个高风险段落，建议优先处理第 2、5、8 段`
- 不建议：`系统已完成深度智能分析，请立即优化以确保通过`

错误提示示例：

- `模型连接失败，请检查 Base URL 或确认本地服务已启动`
- `当前段落过长，系统已自动拆分后重试`
- `导出失败，请稍后重试`

## 14. 开发协作建议

- UI 出图时，每个页面的按钮和状态要标出接口触发点
- 前端开发前，先把 `Document`、`DetectionTask`、`RewriteCandidate` 这些类型固定
- 后端优先实现异步任务接口，否则检测和改写页面很难稳定联调
- 改写对比台一定要先约定差异高亮规则，否则 UI 和前端容易理解不一致
